# Clean URLs and server resolution

This document explains **why** some URLs do not show the `.html` extension and how that relates to the static build and the web server. It is a conceptual explanation: file convention plus server rules, not magic.

## The essentials: mini-astro generates real HTML

If you have this file:

```
src/pages/about.html
```

mini-astro will build:

```
dist/about/index.html
```

(So the URL is `/about` or `/about/`, not `/about.html`.) The path on disk matches the relative path inside `src/pages/`, but the output layout uses directory index for clean URLs.

## How you get “clean” URLs (without .html)

**Clean URLs** (e.g. `/about` instead of `/about.html`) are not a modern framework invention. They rely on standard web server behaviour that has existed for decades.

### Directory index resolution

Many servers (Apache, Nginx, etc.) treat a request for:

```
/about/
```

or, depending on configuration:

```
/about
```

by looking for an index file in that path, typically:

```
/about/index.html
```

That behaviour is called **directory index resolution**. In practice:

1. The browser requests: `https://site.com/about`
2. The server responds with the content of `/about/index.html` (or `/about/` → `index.html` in that path).
3. The URL shown in the browser remains `/about` (or `/about/`).

The `.html` extension does not appear in the address bar.

### Two common ways to get that result

- **Folders with `index.html`**  
  The build writes `dist/about/index.html`. The URL `/about` or `/about/` is resolved to that file via the directory index rule.

- **Flat file and rewrite**  
  The build writes `dist/about.html` and the server has a rewrite rule: “when `/about` is requested, serve `about.html`”. The URL still shows as `/about`.

In both cases, the “clean” URL is a **combination of what the build produces and how the server is configured**.

## Current mini-astro behaviour

mini-astro generates **clean URLs** by default:

- **Root**: `src/pages/index.html` → `dist/index.html` → URL `/`.
- **Other pages**: `src/pages/foo.html` → `dist/foo/index.html`, so the URL is `/foo` or `/foo/`. The server (dev server or static host) resolves via **directory index** (`/foo/` → `foo/index.html`).
- The **dev server** also accepts `/foo` without a trailing slash and serves `foo/index.html` (or `foo.html` in older builds).

So you navigate to `/cookies` and not `/cookies.html`. In production, any host that resolves directory indexes (Nginx, Apache, Netlify, Vercel, etc.) serves these URLs without extra configuration.

## Why clean URLs matter

- **Aesthetics**: `/about` is often perceived as cleaner than `/about.html`.
- **Application-style routes**: consistent with SPAs and frameworks that hide the extension.
- **Technology-agnostic**: the URL does not reveal whether the page is `.html`, `.astro` or generated by another engine. If you change the backend or generator, you can keep the same URLs and avoid broken links.

That makes clean URLs a useful **architectural property** as the project grows or migrates stack.

## Flow summary

```
file in src/pages (e.g. about.html)
        ↓
   mini-astro build
        ↓
   static HTML (dist/about/index.html)
        ↓
   Server: / → index.html; /route → per rules (literal or rewrite/directory index)
        ↓
   Visible URL: with or without .html depending on build + server
```

The web is curious: many “modern” URL innovations are old tricks packaged well.
